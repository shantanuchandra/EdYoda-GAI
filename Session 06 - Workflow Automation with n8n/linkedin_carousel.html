How it works

The 4-node HITL pattern.

1
Trigger — something happens
An email arrives. A form submits. A schedule fires. The workflow wakes up.
2
HTTP Request — the agent classifies it
Your AI agent reads the input and returns a structured decision: standard or escalate, with a draft response.
3
IF node — routes on the decision
Standard → auto-handled at machine speed. Complex → goes to the Wait node.
4
Wait node — the machine pauses for a human
Not cancelled. Not timed out. Holding — until a human taps a button. This is the node most builders skip.

1
Toolkit · Defence 1

The classifier prompt that decides what's complex.

Paste into your HTTP Request body
Classify as COMPLEX (human_gate: true) if ANY of:
— Dietary restriction mentioned
— Lead time under 72 hours
— Budget outside your standard range
— High-risk, irreversible, or unusual request

Return: { "human_gate": true/false, "reason": "..." }
The classifier is the only thing you customise per domain. The Wait node wiring stays identical.

Toolkit · Defences 3 + 4
3
The rejection branch is not optional.
When the manager taps ❌, the workflow must do something — re-draft, escalate, notify. A rejection that goes nowhere is a failure mode. Wire it before you go live.
4
Debug with three questions, not by guessing.
When a workflow breaks: (1) which node is red? (2) what did it receive? (3) what did it expect? Answer all three before touching anything. The error is almost always in the mismatch between 2 and 3.
+
Pin data. Stop flooding your inbox.
After your first successful run, pin the Trigger's output. Every subsequent test replays the pinned email — no real sends, no OAuth re-auth, no test inbox clutter.
