Session 07 · Part 02·Build A · the move
Retry on Fail
Build A · the one checkbox

One checkbox saves the weekend.

✓

Retry on Fail · ON · 3 tries · 2000ms between

Without it: one network blip = one lost customer. With it: n8n retries silently, you never hear about it.

  1. Click the HTTP Request node → URL → paste the new stable URL (your facilitator shared it).
  2. Open the Settings tab on the same node.
  3. Toggle Retry on Fail → ON. Max Tries 3. Wait Between Tries 2000.
  4. Execute step. Publish · version v2 — Stable backend + retry hardening.

Creating a Telegram bot
Build B · BotFather

Three messages with BotFather → a bot exists.

  1. Open Telegram → search @BotFather → tap the verified one.
  2. Type /newbot. BotFather asks for a display name.
  3. Display name: Lumière Ops Notifier.
  4. Username: must be globally unique, must end in Bot. Append your initials.
  5. BotFather replies with a link and the HTTP API token — copy it.
  6. Tap the link → tap Start in the bot's chat.
  7. Open api.telegram.org/bot<TOKEN>/getUpdates → find chat.id.

Session 07 · Part 02·Build C · setup
Customer-facing bot
Build C · second bot

One bot for ops. A different one for customers.

Same BotFather flow as Build B, but a brand-new bot:

  1. @BotFather → /newbot.
  2. Name: Lumière Bakery — Order Bot.
  3. Username: LumiereOrders_<initials>Bot.
  4. Copy the new token (different bot = different token).
  5. In n8n: add credential Lumière Customer Bot.
  6. Tap the bot link → tap Start.

Why two bots?

The ops notifier talks to you. The customer bot talks to customers.

Separating them means you can't accidentally send a customer your internal notification format — and you can revoke either bot's token without affecting the other.

A small upfront discipline that pays back the first time someone leaks a token.
