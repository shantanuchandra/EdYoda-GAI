Sprint 02·How Retrieval Finds
Generative AI for Non-Coders
Three customer queries · two retrieval lanes

It matches meaning, not the exact words.

"no-egg option for cakes?"
Keyword ✕ "no-egg" doesn't appear in the doc.
Semantic ✓ Matches "eggless" by meaning.
"deliver near Versova?"
Keyword ✕ "Versova" sits inside an address line — wrong context for retrieval.
Semantic ✓ Finds the Andheri West delivery zone.
"pincode 400058 deliverable?"
Keyword ✓ Exact digit match on the Andheri zone.
Semantic ~ May find nearby zones — the digits aren't "meaning."
Different blind spots in each lane. Hybrid runs both at once — meaning catches the phrasing, keyword catches the codes. That's Build E.

Sprint 04·The RAG Dial

Types of RAG · turn it up as the knowledge gets bigger

RAG is a dial, not a switch.

L0No retrieval L1CAG L2 — Today's defaultVanilla RAG L3Hybrid RAG L4Agentic RAG
L0
No retrieval
Ask the cake question with nothing loaded → it guesses a lead time. The baseline.
L1 · Build A
CAG
Load whole Lumiere_KB.md into context. No vector DB. Small, stable knowledge.
L2 · B–D
Vanilla RAG
500 locations, 10,000-page manual — retrieve only the matching sections. ~80% of cases.
L3 · Build E
Hybrid RAG
Keyword + semantic fire together. "Clause 3" and "eggless cake" both land.
L4 · S5
Agentic RAG
The agent decides when to retrieve, across multiple sources. Concept today.
Words you'll hear, not today's build —   GraphRAG: retrieves over a map of how entities connect — for cross-document synthesis, where plain RAG is for precise single-fact lookup.  ·  Re-ranking RAG: retrieve twenty pieces, a second model re-sorts to the best three before answering — for precision-critical work.

Build · Ground·Make it cite

B
Build B · Make it cite · ~8 min

Make it show its work.

One short instruction turns "trust me" into "here's where I found it." Every answer comes back with a receipt — the section it pulled from.

Add to your Project instructions: "You are a knowledge agent for Lumière Bakery. Answer questions using ONLY the document(s) in your knowledge. After every answer, name the section, heading, or page you pulled it from, like this: (Source: <section/heading>). Do not add outside facts." → Re-ask. Confirm it ends with a real (Source: …) line.

Build · Ground·Make it refuse
Generative AI for Non-Coders
C
Build C · Make it refuse · ~4 min

Teach it to say "I don't know."

Ask something your doc doesn't cover — watch it guess. Then add one line. Now it refuses. The boundary between a grounded answer and an honest refusal is the difference between a demo and something safe to deploy. Especially on allergens — never guess.

First, ask your persona's out-of-scope question. Watch it guess. Then add to your instructions: "If the answer is not in the document, say: 'I don't have that in my knowledge base' — and stop. Never guess, never fill in from outside knowledge." → Re-ask. Confirm it now refuses instead of guessing.

Build · Ground·Tune the recall

D
Build D · Tune the recall · ~12 min

Break it on purpose, then make it findable.

The agent is only as good as the document you feed it. Today's demo: a wall-of-text policy doc that breaks retrieval, then the same content structured with headings and tags — and the query lands. The model didn't get smarter. The document got findable.

Watch the facilitator demo: → MESSY policy doc: ask a cancellation-rules query. Note what the agent gets wrong or misses. → CLEAN policy doc: same query, structured doc. Note how it lands and cites correctly. Fill in your KB Design Sheet: → What chunk strategy did the clean doc use? → What section tags made the difference? Then test your own Lumière agent: → Ask a question you think it should answer. → If it misses — add a heading or tag to that section, re-upload.

Build · Ground·Stress & ship
Generative AI for Non-Coders
E
Build E · Stress & ship · ~14 min

Three probes, then finalize.

The last test is the real world being awkward — wrong words, adjacent questions, exact codes. Attack it three ways, then lock and ship.

Synonym probe — ask in different words than the doc uses. → "no-egg option" when the doc says "eggless." → Confirm semantic still finds it. Out-of-scope probe — ask something adjacent but absent. → Confirm the refusal still holds. Exact-code probe — ask for a pincode, clause, or section number. → "What pincodes does the Andheri branch cover?" → Keyword fires on the code; meaning handles the prose: hybrid. Lock grounding + citation + refusal. Shipped.

Deploy·RAG or fine-tuning?
Generative AI for Non-Coders
The decision matrix · what changes vs. what stays

RAG vs fine-tuning, in one grid.

What you're changing → FACTS What you're changing → BEHAVIOR
Static
(rarely changes) ↑ Dynamic
(changes often) ↓
Either works Static facts Lumière's founding story, French-technique baking principles. RAG fits; fine-tune fits — pick the cheaper one.
Fine-tune Static behavior House voice. Always reply in the brand's warm-formal tone. JSON output every time. Train it once.
RAG ← Today Dynamic facts This week's offers, new Gurgaon branch, allergen updates, delivery zones. Update the document — never retrain the model.
Both (hybrid) Dynamic behavior Rare. A bot whose tone must adapt per customer. RAG for context, fine-tune for voice scaffolding.
One honesty line: fine-tuning does not remove being wrong — it makes the model wrong in your brand voice. Grounding is what gives an answer a source. And this rule is practitioner consensus, not settled law — expect honest debate at the edges.
